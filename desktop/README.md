# Mytronome desktop & Android (Tauri)

The Tauri shell that wraps the `web/` SPA into native desktop (Windows/macOS/Linux)
and Android apps. The Rust side lives in `src-tauri/`; the native Google Drive
authorization plugin for Android is in `tauri-plugin-google-drive-auth/`.

`src-tauri/gen/` is generated by Tauri and **git-ignored** — it is regenerated by
`tauri android init` and rebuilt on every build. Nothing generated is committed;
the only thing that has to survive a regenerate is the release-signing delta,
documented below.

## Building

```sh
# Desktop (from repo root)
npm run tauri -w desktop -- dev            # dev, hot reload
npm run tauri -w desktop -- build          # release bundles

# Android — needs ANDROID_HOME + an NDK (Tauri auto-detects the installed NDK)
npm run tauri -w desktop -- android init            # (re)generate src-tauri/gen/android
npm run tauri -w desktop -- android dev             # install + run on device/emulator (debug)
npm run tauri -w desktop -- android build --apk --target aarch64   # signed release APK, arm64
npm run tauri -w desktop -- android build --aab                    # release App Bundle (all ABIs)
```

Debug and release use different application IDs (`ca.dovall.mytronome.debug` vs
`ca.dovall.mytronome`, via `debugApplicationIdSuffix` in `tauri.conf.json`), so
both install side by side.

> On Windows, `tauri android build` symlinks the compiled `.so` into `jniLibs/`.
> That needs **Developer Mode** on (Settings → System → For developers) or an
> elevated terminal, otherwise it fails with _"Creation symbolic link is not
> allowed for this system."_ A first release build can also flake once on a
> cross-drive `IllegalArgumentException: ... different roots` (the Kotlin daemon,
> when the Cargo registry and project are on different drives) — just re-run.

## Android release signing

Release builds are signed from a keystore that is **never** committed. Two pieces
make it work, and both must be recreated on a fresh checkout or after a
`tauri android init` (which regenerates `app/build.gradle.kts`):

### 1. The keystore

A PKCS12 keystore is the app's permanent signing identity. It lives **outside the
repo** and must be backed up — if it's lost you can't ship an update under the same
signature.

- **Location:** `C:\Users\tyrda\keys\mytronome-release.jks` (back this up off-machine)
- **Alias:** `mytronome`
- **Password:** in the password manager (store and key password are the same)
- **Regenerate (only if starting over):**

  ```sh
  keytool -genkeypair -v -storetype PKCS12 \
    -keystore <path>/mytronome-release.jks -alias mytronome \
    -keyalg RSA -keysize 2048 -validity 10000 \
    -dname "CN=Mytronome, O=Dovall, C=CA"
  ```

The signing certificate's **SHA-1** must be registered on the _release_ Android
OAuth client in Google Cloud Console (package `ca.dovall.mytronome`, no `.debug`),
or native Drive sign-in fails at runtime with `APIException: 10 (DEVELOPER_ERROR)`.
Read the SHA-1 with `keytool -list -v -keystore <path> -alias mytronome`, or from a
built APK with `apksigner verify --print-certs <apk>`.

### 2. `keystore.properties`

Create `src-tauri/gen/android/keystore.properties` (git-ignored via
`src-tauri/.gitignore`). It points the build at the keystore and holds the
passwords — local only, never committed:

```properties
storeFile=C:/Users/tyrda/keys/mytronome-release.jks
keyAlias=mytronome
storePassword=<keystore password>
keyPassword=<keystore password>
```

### 3. The signing block in `app/build.gradle.kts`

`tauri android init` regenerates `src-tauri/gen/android/app/build.gradle.kts` from
Tauri's template, wiping any edits. After a regenerate, re-append this block
**verbatim** to the end of that file (after the `apply(from = "tauri.build.gradle.kts")`
line). It creates its own scope and only touches the `android {}` accessor, so it
doesn't depend on the template's _structure_ (buildTypes layout, etc.) and won't rot
when Tauri reshuffles that. Its one dependency is the `import java.util.Properties`
at the top of the generated file — which the template always emits because it uses
`Properties` itself (`tauriProperties`). Do **not** fully-qualify it as
`java.util.Properties`: in a Gradle Kotlin DSL script `java` resolves to the Java
plugin accessor, not the package root, so it fails with `Unresolved reference: util`.
When `keystore.properties` is absent the block does nothing (release stays unsigned,
debug is unaffected), so a fresh checkout still builds.

```kotlin
// --- Release signing (Mytronome) -------------------------------------------
// See desktop/README.md → "Android release signing". Re-append verbatim after
// `tauri android init` regenerates this file.
run {
    val keystorePropertiesFile = rootProject.file("keystore.properties")
    if (keystorePropertiesFile.exists()) {
        val keystoreProperties = Properties().apply {
            keystorePropertiesFile.inputStream().use { load(it) }
        }
        android {
            signingConfigs {
                create("release") {
                    keyAlias = keystoreProperties.getProperty("keyAlias")
                    keyPassword = keystoreProperties.getProperty("keyPassword")
                    storeFile = file(keystoreProperties.getProperty("storeFile"))
                    storePassword = keystoreProperties.getProperty("storePassword")
                }
            }
            buildTypes.getByName("release").signingConfig =
                signingConfigs.getByName("release")
        }
    }
}
```

### Verifying a build is signed

```sh
apksigner verify --verbose --print-certs \
  src-tauri/gen/android/app/build/outputs/apk/universal/release/app-universal-release.apk
```

Expect `Verified using v2 scheme ... : true` and
`Signer #1 certificate DN: CN=Mytronome, O=Dovall, C=CA`. (v1/JAR signing is
correctly absent — `minSdk = 24`, so the v2 scheme covers every supported device.)

## App icons

The master is `src-tauri/app-icon.svg` (the filled tile). `npm run gen:icon`
rasterizes it to a 1024×1024 `app-icon.png` (via `@resvg/resvg-js`, gitignored)
and runs `tauri icon`, which writes:

- **committed:** the desktop set + `icons/ios/` under `src-tauri/icons/`;
- **into the gitignored `gen/` project:** the Android `mipmap-*` launcher icons.

Because the Android icons land in `gen/` (which `tauri android init` regenerates
from Tauri defaults), **re-run `npm run gen:icon` after any `android init`** to
restore the launcher icon — same post-regenerate ritual as the signing block.
The web favicon is a separate hand-authored file, `web/public/favicon.svg`
(theme-adaptive via `prefers-color-scheme`).
